## Why this matters

Five small, independent dependency defects, each verified. None is urgent
alone; together they are a 20-minute cleanup that removes a supply-chain
surface, makes the build reproducible instead of luck-of-the-resolver, and
fixes an import that fails in the published wheel:

1. **Unused runtime dep.** `pydantic-settings>=2.0` is declared as a runtime
   dependency but never imported anywhere in `src/`, `tests/`, or `scripts/` —
   every install pays for a library the code doesn't use.
2. **`fastmcp` floor spans a breaking major.** The floor is `>=2.12` with no
   ceiling, but the code imports 3.x-only paths (`fastmcp.server.middleware`,
   `fastmcp.tools.base.ToolResult`) and the lock resolves `3.4.0`. A clean
   resolve on another machine could legally pick a 2.x release where those
   imports don't exist — the build works only because the lock happens to pin
   3.4.0.
3. **`jsonschema` is dev-only but imported by shipped code.** `src/codescent/
   evals/agent_ux/schemas.py:13` does `import jsonschema`, but `jsonschema` is
   declared only under `[dependency-groups].dev`. The wheel ships `evals/` (the
   hatch build packages all of `src/codescent`), so importing
   `codescent.evals.agent_ux.schemas` from an installed wheel raises
   `ModuleNotFoundError`.
4. **`fff-search` core dep on a loose pre-1.0 pin.** `>=0.9.6` with no ceiling:
   a `0.x` library treats minor bumps as allowed to break, so `0.10`/`1.0` could
   resolve into a fresh install and break the primary search backend.
5. **`pyahocorasick` declared twice** — in the `fast` optional-extra and again
   in the dev group, two sources of truth for one pin.

## Steps

### Step 1: Remove the unused `pydantic-settings` dependency

Delete the `"pydantic-settings>=2.0",` line from `[project].dependencies`.

**Verify**:
- `grep -rniE 'pydantic_settings|BaseSettings' src tests scripts` → no matches
  (confirms nothing needs it).
- `uv lock && uv sync` → exit 0.
- `uv run python -c "import codescent; import codescent.mcp.server; import codescent.services.config"` → exit 0.

### Step 2: Pin the `fastmcp` and `fff-search` floors

- `fastmcp`: change `>=2.12` to `>=3,<4`. The code already uses 3.x-only import
  paths, so the real floor is 3.x; the ceiling prevents a future 4.x from
  silently breaking those imports.
- `fff-search`: change `>=0.9.6` to `>=0.9.6,<0.10` (treat pre-1.0 minor bumps
  as breaking for a core backend).

**Verify**:
- `uv lock && uv sync` → exit 0, and the resolved `fastmcp` is still 3.x
  (`uv tree --depth 1 | grep fastmcp`), `fff-search` still 0.9.x.
- `uv run pytest tests/contract -q` → pass (the MCP surface still imports
  cleanly against the pinned fastmcp).

### Step 3: Fix the `jsonschema` / wheel-packaging mismatch

First determine whether runtime code (outside `evals/`, `smoke/`, `tests/`)
imports `codescent.evals`:

```
grep -rn "codescent.evals\|from codescent import evals\|import codescent.evals" src --include=*.py | grep -v "src/codescent/evals/"
```

- **If that returns nothing** (expected — `evals/` is a dev/CI harness): exclude
  the harness from the wheel so `jsonschema` need not ship. Add to
  `[tool.hatch.build.targets.wheel]`:

  ```toml
  exclude = ["src/codescent/evals", "src/codescent/smoke"]
  ```

  (Confirm the exact `smoke` package path first: `ls src/codescent/smoke`.)

- **If runtime code *does* import `codescent.evals`**: do not exclude it;
  instead move `jsonschema>=4.0` out of `[dependency-groups].dev` and declare an
  optional extra for the eval harness, e.g.
  `[project.optional-dependencies] evals = ["jsonschema>=4.0", "fastmcp>=3,<4"]`,
  and note that the harness requires `uv sync --extra evals`.

**Verify**:
- `uv build` → produces `dist/*.whl`.
- If you excluded evals: `unzip -l dist/*.whl | grep -c 'codescent/evals'` → `0`,
  and `unzip -l dist/*.whl | grep -c 'codescent/mcp/server'` → `1` (runtime
  still shipped).
- `uv run pytest tests/ -q -k "eval or agent_ux" ` → pass (the dev environment
  still has `jsonschema`, since `uv sync` installs the dev group; excluding from
  the *wheel* does not affect the dev env).

### Step 4: De-duplicate `pyahocorasick`

Remove `"pyahocorasick>=2.0",` from `[dependency-groups].dev`. The `fast` extra
already declares it; make the dev group depend on the extra instead so the pin
has one source of truth — add `"codescent[fast]"` to the dev group if `uv`
resolves a self-referential extra cleanly; if it does not, leave the dev entry
but add a comment that it must match the `fast` extra's pin.

**Verify**:
- `uv lock && uv sync` → exit 0.
- `uv run python -c "import ahocorasick"` → exit 0 (the accelerator is still
  installed in the dev env).
- `grep -c 'pyahocorasick' pyproject.toml` → `1` (single declaration) if the
  `codescent[fast]` reference worked; otherwise `2` with an explanatory comment.

### Step 5: Full green

**Verify**:
- `uv run pytest` → no new failures.
- `uv run ruff check .` / `uv run basedpyright` → clean.
- `git diff --stat` shows only `pyproject.toml` and `uv.lock` changed.
